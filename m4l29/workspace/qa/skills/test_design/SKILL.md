# test_design：人事系统测试设计规范

## 测试策略总则

### 测试四象限（优先级从高到低）

| 象限 | 覆盖内容 | 不通过的含义 |
|------|---------|------------|
| Q1 核心流程 | M1 入职审批链 + M2 离职审批链 Happy Path | ❌ 阻断发布 |
| Q2 权限矩阵 | 所有角色 × 所有模块 × 所有操作的交叉验证 | ❌ 阻断发布 |
| Q3 审批节点 | 每个审批动作（通过/驳回/撤回/超时升级）| ❌ 阻断发布（涉及审批的模块） |
| Q4 边界异常 | 空字段/格式校验/日期非法/并发/超长输入 | ⚠️ 影响发布 |

### 测试设计原则

1. **一个测试用例只验证一件事**：不把"入职审批通过"和"合同生成"写成一个用例
2. **角色身份是测试上下文的一部分**：每个用例开头注明测试角色
3. **审批流测试必须覆盖全链路**：提交→待办生成→审批通过→下一节点→最终状态
4. **权限测试必须覆盖所有角色**：一条权限规则 × N 个角色 = N 条测试用例

## M1 入职管理 测试设计

### 核心流程
1. HR 登录 → 录入新员工信息 → 提交入职申请 → 系统生成工号 → 待办推送到直属经理
2. 直属经理登录 → 查看待办 → 审批通过 → 待办推送到管理员
3. 管理员登录 → 查看待办 → 审批通过 → 合同自动生成 → 账号自动开通 → 员工状态变为"在职"

### 必测字段（信息录入）
| 字段 | 类型 | 规则 | 测试点 |
|------|------|------|--------|
| 姓名 | string | 必填，2-50字符 | 空/1字符/50字符/51字符/特殊字符 |
| 身份证号 | string | 必填，18位，格式校验 | 空/17位/18位非法校验码/19位/X结尾 |
| 手机号 | string | 必填，11位，1开头 | 空/10位/12位/非1开头/含字母 |
| 邮箱 | string | 必填，合法邮箱格式 | 空/无@/无域名/含空格 |
| 入职日期 | date | 必填，不能早于出生日期 | 空/等于出生日期/早于出生日期/未来日期(>30天) |
| 部门 | select | 必填，从组织架构选择 | 空/不存在的部门ID |
| 直属上级 | select | 必填，在职员工 | 空/已离职员工/自己 |

### 审批链测试
```
发起人(HR) → 直属经理(审批/驳回) → 管理员(审批/驳回)
     ↑__ ← 驳回回到发起人
```
- 每个节点审批通过 → 状态正确流转
- 驳回 → 回到 HR，附驳回原因，HR 可修改后重新提交
- 撤回 → HR 在直属经理审批前可撤回，状态还原
- 超时升级 → 直属经理 48h 未审批 → 自动升级到管理员

### 合同生成验证
- 审批链全部通过后，合同自动生成（含员工姓名、部门、岗位、入职日期、薪资）
- 合同模板字段无缺失

## M2 离职管理 测试设计

### 核心流程
1. 员工/HR 登录 → 提交离职申请（离职原因、最后工作日）
2. 直属经理审批 → 确认最后工作日 → 分配工作交接人
3. 工作交接人确认 → 确认已接收全部工作文档和待办
4. 资产管理员确认 → 电脑/工牌/门禁卡等已归还
5. 离职审批最终通过 → 账号注销 → 员工状态变为"已离职"

### 必测检查点
| 阶段 | 检查项 |
|------|--------|
| 离职申请 | 离职原因必填、最后工作日 ≥ 今天 + 30天（提前期） |
| 工作交接 | 交接人必须是在职员工、交接确认后方可进入下一节点 |
| 资产归还 | 资产清单完整（电脑/工牌/门禁/其他）、缺失项需备注 |
| 账号注销 | 离职审批通过后 24h 内账号自动注销、数据保留策略 |

### 离职审批链（与入职不同——多 2 个确认节点）
```
发起人(员工/HR) → 直属经理(审批) → 工作交接人(确认) → 资产管理员(确认) → 最终审批
     ↑__ ← 驳回(任一点可驳回)
```

## M3 员工信息管理 测试设计

### 权限矩阵（核心测试点）

| 操作 | HR | 员工(自己) | 员工(他人) | 直属经理(下属) | 管理员 |
|------|:--:|:---------:|:---------:|:------------:|:-----:|
| 查看基本信息 | ✅ | ✅ | ❌ | ✅(下属) | ✅ |
| 修改基本信息 | ✅ | ❌ | ❌ | ❌ | ✅ |
| 查看薪资信息 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 批量导入 | ✅ | ❌ | ❌ | ❌ | ✅ |
| 批量导出 | ✅ | ❌ | ❌ | ❌ | ✅ |
| 岗位变动 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 部门调动 | ❌ | ❌ | ❌ | ❌ | ✅ |

### 边界测试
- 批量导入 0 条 / 1 条 / 1000 条 / 10000 条
- 批量导入含格式错误行 → 跳过错误行并报告，不中断正常行
- 员工状态变更（在职→离职）后，其作为"直属上级"的引用是否清理

## M4 权限与审批流 测试设计

### 权限测试方法

**测试方法**：对每个角色，逐一验证权限矩阵中 ✅ 的操作是否可执行、❌ 的操作是否被阻止。

**关键验证点**：
1. URL 越权：普通员工直接访问 HR 专属页面 URL → 应拒绝
2. API 越权：员工通过 API 请求修改他人信息 → 应拒绝
3. 数据越权：员工查询接口返回结果中是否包含他人数据

### 审批链配置测试
- 审批链节点配置：N 个角色组成审批链，逐一节点流转
- 审批链修改：修改审批链（增加/删除节点）后，新申请按新链、已提交申请按旧链
- 超时参数：修改超时时间后生效

### 审批动作测试
| 动作 | 前置条件 | 预期结果 |
|------|---------|---------|
| 审批通过 | 当前节点待办状态 | 流转到下一节点/最终完成 |
| 审批驳回 | 当前节点待办状态 | 退回发起人，含驳回原因 |
| 撤回 | 已提交 + 未审批 | 状态还原到草稿/已撤回 |
| 超时升级 | 审批人超时未处理 | 自动升级到上级或管理员 |

## 边界与异常通用清单

### 输入边界
- 全部必填字段为空 → 阻止提交
- 字符串字段：1字符 / 最大长度 / 最大长度+1 / SQL 注入字符 / XSS 字符
- 数字字段：0 / 负数 / 最大值 / 最大值+1
- 日期字段：1970-01-01 / 2099-12-31 / 非法格式
- 下拉选择：不存在的选项 ID

### 并发场景
- 两人同时审批同一条待办 → 后者提示"已被处理"
- 两人同时为同一员工发起入职 → 工号唯一性检测

### 状态一致性
- 入职审批通过后 → 员工出现在组织架构
- 离职审批通过后 → 员工从在职列表消失
- 审批中撤回 → 所有节点待办清除
