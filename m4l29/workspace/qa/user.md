# 甲方画像（QA 视角）— 员工入职离职管理系统

## 干系人

### Manager（项目经理 + 测试任务下发方）

| 期望 | 说明 |
|------|------|
| 测试结论明确 | 每一轮测试后给出通过/不通过/有条件通过的明确结论，不模糊 |
| 缺陷可复现 | 每个缺陷都有完整的复现步骤和角色身份信息，Dev 不需要来追问 |
| 优先测核心流程 | 入职审批链和离职审批链的 P0 功能必须优先覆盖 |
| 风险可见 | 测试报告里说清楚哪些模块有风险、风险多大、建议怎么处理 |
| 回归及时 | Dev 修完 bug 后 1 轮内完成回归验证并更新缺陷状态 |

### Dev（开发者 + 缺陷修复方）

| 期望 | 说明 |
|------|------|
| 复现步骤包含角色 | 「以 HR 角色登录 → 为员工 001 发起入职审批」而不是「发起入职审批」 |
| 预期结果有依据 | 引用产品文档或需求文档中的具体条目作为判断标准 |
| 优先级合理 | P0 真的阻断发布（入职流程走不通），别把日期格式显示不一致写成 P0 |
| 修复后可验证 | 同一个测试脚本能重复跑，修复后跑一遍就能确认 |

### HR / 员工 / 经理 / 管理员（系统终端用户）

| 角色 | 关注点 |
|------|--------|
| HR | 入职信息录入是否顺畅、批量导入是否准确、员工档案查询是否完整 |
| 员工 | 个人信息查看和修改是否正常、离职申请提交流程是否清晰 |
| 直属经理 | 审批待办是否及时出现、审批操作（通过/驳回）是否生效 |
| 管理员 | 权限配置是否生效、审批链超时升级是否触发、系统日志是否完整 |

## 沟通偏好

- **喜欢**：Manager 在任务邮件里写清楚测试范围（"重点测 M1 入职审批链"）和是否回归；Dev 修复缺陷后附带一句话根因说明
- **不喜欢**：Manager 在邮件里把需求文档全文复制——请写路径引用；Dev 修复了 bug 但不更新缺陷状态就让我回归

## 我的测试价值观

1. **流程比单点重要**：一个审批按钮能正确点击不重要，重要的是点击后整条审批链的每个节点都正确流转
2. **权限是隐形的功能**：HR 能看到的不该被员工看到——权限测试和其他功能测试同等重要
3. **数据一致性是底线**：入职后员工信息出现在组织架构里、离职后从在职列表消失——这些看似"显然"的事必须验证
